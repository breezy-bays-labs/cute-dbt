# Release discipline

cute-dbt publishes to crates.io from **`v0.1.0+`** via
[`release-plz`](https://release-plz.dev/) + OIDC trusted publishing.
v0.x is **explicitly unstable** per Cargo SemVer convention.

Full policy lives in
[`AGENTS.md` §Release discipline](https://github.com/breezy-bays-labs/cute-dbt/blob/main/AGENTS.md#release-discipline)
in the repo; the operational summary you need as a user:

## Versioning

| Bump | v0.x | v1.0+ |
|---|---|---|
| Patch (`0.1.0 → 0.1.1`) | Bug fix / additive only | Bug fix / additive only |
| Minor (`0.1.0 → 0.2.0`) | **May break** (CLI flag renames, output-shape changes, exit-code changes) | Backward-compatible additive only |
| Major (`1.x → 2.0.0`) | n/a (we're pre-1.0) | Breaking change |

**The "API" under SemVer in v1.0+** = the binary's CLI surface
(`cute-dbt --help`), the exit-code mapping (the `PreflightError` →
ExitCode contract), and the zero-egress proof (the resource-ref lint
+ headless-browser network-block test). The library crate
(`cute4dbt` as a Rust library) is **internal-only** during v0.x;
v1.0 may explicitly promote some of it.

## What this means for you as a consumer

- **`cargo install cute4dbt`** is the canonical install path from
  `v0.1.0+`. The binary is `cute-dbt`.
- **Pin to a minor version** during v0.x (`cute4dbt = "0.1"`) and
  read the CHANGELOG before bumping to `0.2`.
- **CHANGELOG is canonical**:
  [`CHANGELOG.md`](https://github.com/breezy-bays-labs/cute-dbt/blob/main/CHANGELOG.md)
  in the repo, generated by release-plz from conventional commits +
  human-edited in the release PR.
- **Yanks** happen only for **security** or **licensing** issues.
  cute-dbt never amends a published version number; even a one-character
  CLI rename ships as a new patch (or minor in v0.x). Yanked versions
  are documented under a `[YANKED]` heading in CHANGELOG.

## Tag policy

Every published version is a **signed annotated tag** created by
release-plz in CI under OIDC. Tag deletion is forbidden — to "fix" a
bad tag, ship the next patch (per the yank policy above).

## Release cadence

Event-driven via release-plz's auto-PR pattern: on every push to
`main`, release-plz opens (or updates) a release PR proposing the next
version + the CHANGELOG entry. Merging the release PR triggers the
tag + crates.io publish. There is no calendar SLA on v0.x bug-fix
turnaround.

## For contributors

Conventional commit prefixes drive `release-plz`'s version-bump
inference, with Cargo-SemVer-aware mapping:

- **v0.x** (current): `feat` → patch (additive); `fix` → patch;
  `BREAKING CHANGE` footer → minor (the v0.x breaking-change line).
- **v1.0+**: `feat` → minor (additive); `fix` → patch;
  `BREAKING CHANGE` footer → major.
- Non-versioning prefixes (`docs` / `chore` / `test` / `refactor` /
  `ci` / `adr` / `closeout`) do not trigger version bumps.

Manual `cargo publish` is **forbidden** — OIDC + release-plz is the
only publish path. See
[`CONTRIBUTING.md`](https://github.com/breezy-bays-labs/cute-dbt/blob/main/CONTRIBUTING.md#release-discipline)
for the contributor-facing version of this policy.
